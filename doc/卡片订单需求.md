## 卡片订单与升级业务说明

> 本文基于 `t_card_package`（卡片/基金套餐）设计用户下单与升级的整体逻辑，输出订单表结构及核心流程，供后端与前端联调参考。

---

### 1. 业务场景概述

1. 用户可从「卡片商城」购买任意卡片（普通卡、白银卡、白金卡、黑金卡等）。
2. 已购买的卡片支持 **随时补差价升级**：差价 = 目标卡价格 - 当前卡价格。
3. 升级成功后：
   - 立即获得新增算力（目标算力 - 原算力）。
   - 触发直推/间推奖励（计算基于差价金额）。
4. 卡片订单需要覆盖 **首次购买** 与 **补差价升级** 两种场景。

---

### 2. 订单表设计（`t_card_order`）

```sql
CREATE TABLE `t_card_order` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `order_no` VARCHAR(64) NOT NULL COMMENT '订单号',
  `user_id` BIGINT NOT NULL COMMENT '用户ID',
  `user_wallet_id` BIGINT DEFAULT NULL COMMENT '资金账户ID（选填）',

  `order_type` TINYINT NOT NULL DEFAULT 0 COMMENT '订单类型：1-购买新卡，2-补差价升级',
  `pay_token` VARCHAR(32) NOT NULL DEFAULT 'USDT' COMMENT '支付资产（USDT/BPAY...）',
  `pay_amount` DECIMAL(20,6) NOT NULL COMMENT '本次实际支付金额（USDT）',

  `package_id` INT NOT NULL COMMENT '目标卡片ID（指向 t_card_package.id）',
  `package_name` VARCHAR(64) NOT NULL COMMENT '目标卡片名称（冗余）',
  `package_price` DECIMAL(20,6) NOT NULL COMMENT '目标卡片价格（冗余）',
  `package_hashrate` DECIMAL(20,6) NOT NULL COMMENT '目标卡片算力（冗余）',

  `source_card_id` BIGINT DEFAULT NULL COMMENT '升级来源卡片ID（仅 order_type=1 有值）',
  `source_package_id` INT DEFAULT NULL COMMENT '升级前卡片类型',
  `source_hashrate` DECIMAL(20,6) DEFAULT NULL COMMENT '升级前卡片算力',

  `hashrate_increase` DECIMAL(20,6) NOT NULL COMMENT '本次新增算力（购买=目标算力；升级=差值）',
  `status` TINYINT NOT NULL DEFAULT 0 COMMENT '订单状态：0-待支付，1-支付中，2-已完成，3-已取消，4-已退款',
  `pay_tx_hash` VARCHAR(128) DEFAULT NULL COMMENT '链上支付Tx（若需要）',

  `direct_reward_usdt` DECIMAL(20,6) DEFAULT 0 COMMENT '直推奖励金额',
  `direct_reward_hashrate` DECIMAL(20,6) DEFAULT 0 COMMENT '直推奖励算力',
  `indirect_reward_usdt` DECIMAL(20,6) DEFAULT 0,
  `indirect_reward_hashrate` DECIMAL(20,6) DEFAULT 0,

  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `paid_at` DATETIME DEFAULT NULL COMMENT '支付时间',
  `completed_at` DATETIME DEFAULT NULL COMMENT '完成时间',
  `remark` VARCHAR(255) DEFAULT NULL COMMENT '备注',

  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_no` (`order_no`),
  KEY `idx_user_id` (`user_id`),
  KEY `idx_status` (`status`),
  KEY `idx_source_card_id` (`source_card_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='卡片订单表（购买/升级）';
```

### 3. 升级日志表（`t_card_upgrade_log`）

> 由于业务允许「逐级升级」或「直接跨级补差价」，单条订单无法完整记录一位用户的历史升级链路，因此需要额外使用升级日志表追踪每一次等级跃迁，便于审计、风控与报表。

```sql
CREATE TABLE `t_card_upgrade_log` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `order_id` BIGINT NOT NULL COMMENT '关联订单ID（t_card_order.id）',
  `user_id` BIGINT NOT NULL COMMENT '用户ID',
  `card_id` BIGINT NOT NULL COMMENT '被升级的用户卡片ID',

  `from_package_id` INT NOT NULL COMMENT '升级前卡片类型',
  `from_package_name` VARCHAR(64) NOT NULL COMMENT '升级前卡片名称',
  `from_price` DECIMAL(20,6) NOT NULL COMMENT '升级前卡片价格',
  `from_hashrate` DECIMAL(20,6) NOT NULL COMMENT '升级前卡片算力',

  `to_package_id` INT NOT NULL COMMENT '升级后卡片类型',
  `to_package_name` VARCHAR(64) NOT NULL COMMENT '升级后卡片名称',
  `to_price` DECIMAL(20,6) NOT NULL COMMENT '升级后卡片价格',
  `to_hashrate` DECIMAL(20,6) NOT NULL COMMENT '升级后卡片算力',

  `diff_price` DECIMAL(20,6) NOT NULL COMMENT '本次补差价金额',
  `diff_hashrate` DECIMAL(20,6) NOT NULL COMMENT '本次获得的新增算力',

  `upgrade_path` VARCHAR(32) DEFAULT NULL COMMENT '升级路径标识，例如 NORMAL->DIAMOND',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `remark` VARCHAR(255) DEFAULT NULL COMMENT '备注',

  PRIMARY KEY (`id`),
  KEY `idx_order_id` (`order_id`),
  KEY `idx_user_id` (`user_id`),
  KEY `idx_card_id` (`card_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='卡片升级日志';
```

**使用说明**

- 若用户分多次升级（普通→白银→白金→黑金），则会产生多条日志，每条日志对应一次订单；
- 若用户一次性从普通卡跨级升级到黑金卡，也只需要一条订单 + 一条日志，`from_package_id=普通卡`、`to_package_id=黑金卡`，`diff_price=黑金-普通`；
- 通过 `card_id + created_at` 可以还原某张卡片完整的升级轨迹；
- 后台可以根据升级日志做数据分析，例如统计不同等级的升级次数、补差价金额等。

### 4. 订单生命周期

1. **创建订单**
   - 用户选择目标卡片，计算应付金额（新购=目标价；升级=差价）。
   - 生成 `order_no`，写入 `t_card_order`，状态=0。
2. **支付与校验**
   - 扣除用户资金（USDT）。
   - 成功后更新 `status=2`，记录 `pay_amount`、`paid_at`。
3. **发放算力**
   - `hashrate_increase` 写入用户卡片表 / 算力统计表。
   - 若是升级，同步更新原卡片记录的等级/算力。
4. **触发推荐奖励**
   - 找到直推、间推人，计算 10%/5% 的 USDT 与算力奖励，写入奖励表。
5. **更新小区业绩**
   - 本次 `pay_amount` 计入推荐链上所有上级的“团队业绩”。
   - 同时刷新其“小区业绩”（去掉最大线后的总和）。

---

### 5. 关键字段说明

| 字段 | 场景 | 说明 |
|------|------|------|
| `order_type` | 0/1 | 区分购买与升级。升级时必须关联 `source_card_id` |
| `pay_amount` | 新购 / 升级 | 新购=目标价格；升级=差价 |
| `hashrate_increase` | 新购 / 升级 | 新购=目标算力；升级=目标算力-原算力 |
| `package_*` 冗余字段 | 所有 | 避免历史数据受套餐调整影响 |
| `direct_reward_*` | 奖励 | 仅用于日志和审计，真实奖励写入奖励表 |

### 6. 升级流程（伪代码）

```pseudo
1. 前端：选择已有卡片 + 目标卡片 -> 请求后端计算差价
2. 后端：
   - verify(user_card exists && target > current)
   - diff_price = target_price - current_price
   - diff_hashrate = target_hashrate - current_hashrate
   - 创建订单(order_type=1, pay_amount=diff_price, hashrate_increase=diff_hashrate)
3. 用户支付 -> 更新订单状态
4. 成功后：
   - 更新用户卡片等级、算力
   - 派发奖励
   - 更新业绩 / 等级
```

---

### 7. 与其他表的关系

- `t_card_package`：提供 `package_id/price/hashrate` 等基础信息。
- `t_user_card`（或类似表）：记录用户当前持有卡片，用于升级判断。
- `t_user_level_config`：升级后更新用户小区业绩 & 等级判定。
- `t_reward_log`：记录直推 & 间推奖励。
- `t_card_upgrade_log`：用于审计每一次升级行为，是 `t_card_order` 的补充。

### 8. 接口参考（简版）

1. `POST /api/card-orders/price`
   - 入参：`userCardId?`, `targetPackageId`
   - 出参：差价、算力增量、可用余额等
2. `POST /api/card-orders`
   - 创建购买/升级订单
3. `POST /api/card-orders/{orderNo}/pay`
   - 处理支付结果（或监听链上回调）
4. `GET /api/card-orders/my`
   - 分页查询我的订单
